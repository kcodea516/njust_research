# 车牌检测与识别模块宏观方案

## 1. 模块目标

车辆模块已经解决了下面两个问题：

```text
车辆在哪里
车辆属于哪一类货车
```

车牌模块要解决的是：

```text
车牌在哪里
车牌号码是什么
如果无法可靠识别，能否给出低置信度或无法识别的判断
```

因此系统最终不是一个模型完成全部任务，而是多个模块串联：

```text
输入高速路图片
  -> 车辆检测与车型分类
  -> 车牌检测
  -> 车牌裁剪与增强
  -> OCR 字符识别
  -> 结构化输出
```

最终输出可以设计成：

```json
{
  "vehicle_type": "Zhatu",
  "vehicle_conf": 0.91,
  "vehicle_box": [x1, y1, x2, y2],
  "plate_box": [x1, y1, x2, y2],
  "plate_text": "黄A0A330",
  "plate_conf": 0.86,
  "plate_status": "recognized"
}
```

如果车牌太小、过曝、反光或人眼也无法分辨，则输出：

```json
{
  "plate_text": null,
  "plate_conf": 0.18,
  "plate_status": "unreadable",
  "reason": "车牌过曝或分辨率不足"
}
```

这比强行输出一个错误车牌更合理。

## 2. 为什么要分成两层

车牌识别不建议直接在整张高速路图片上做 OCR，因为图片里车辆大、车牌小，背景复杂，且有夜间、反光、模糊、斜视角等问题。

更稳的方式是两层：

```text
第一层：车牌检测
  输入：整图或车辆区域
  输出：车牌框

第二层：车牌 OCR
  输入：裁剪后的车牌小图
  输出：车牌字符
```

这样做的好处是：

```text
车牌检测只负责找位置
OCR 只负责识别清晰裁剪图
每一层都可以单独评估和改进
错误来源更容易分析
```

## 3. 当前数据的主要困难

当前项目中的高速路货车图存在几个明显难点：

```text
车牌在整图中占比很小
夜间灯光导致车牌过曝
黄牌反光严重
拍摄角度斜，车牌有透视变形
部分图片中车牌人眼也无法看清
现有 chexing 数据集没有车牌框标注，也没有车牌文本标注
```

这意味着：

```text
没有车牌标注，就无法直接训练车牌检测模型
没有车牌文本，就无法监督训练 OCR 模型
完全不可见的车牌，模型也不能可靠恢复真实字符
```

因此车牌模块不能直接从现有 chexing 数据集自然推进，必须补充数据或降低目标。

## 4. 数据集应该怎么构造

车牌模块需要两类标注。

### 4.1 车牌检测数据

车牌检测数据只需要标注车牌框。

YOLO 格式可以设计为单类别：

```text
class 0 = plate
```

目录结构可以是：

```text
data/yolo_plate_dataset/
  images/
    train/
    val/
  labels/
    train/
    val/
  plate.yaml
```

每个标签文件内容类似：

```text
0 x_center y_center width height
```

这一层不需要知道车牌号码，只要知道车牌在哪里。

### 4.2 车牌 OCR 数据

OCR 数据需要车牌裁剪图和对应文本。

可以整理成：

```text
data/plate_ocr_dataset/
  images/
    train/
    val/
  labels.csv
```

`labels.csv` 类似：

```csv
image,text
000001.jpg,黄A0A330
000002.jpg,黄B12345
```

OCR 的训练难度比检测更高，因为需要准确文本标注。

## 5. 如果没有自己的车牌标注怎么办

如果暂时没有自己的车牌框和文本标注，车牌模块可以分阶段推进。

### 5.1 第一阶段：使用公开数据集和预训练模型做 baseline

先使用公开中文车牌数据集或现成车牌识别模型，跑一版 baseline。

目标不是马上达到高精度，而是回答：

```text
现成车牌模型在高速货车图上能识别多少
哪些情况失败
失败主要来自检测不到，还是 OCR 读错
```

这一步可以作为项目中的“基线系统搭建”。

### 5.2 第二阶段：少量人工标注真实高速货车图

从 overload 数据集和 chexing 数据集中抽取一部分图片，人工标注：

```text
车牌框
是否可读
如果可读，标注车牌文本
```

建议先小规模做：

```text
100-300 张车牌框标注
其中可读车牌再标注文本
```

这样就能做领域适配，而不是完全依赖公开数据集。

### 5.3 第三阶段：训练车牌检测模型，OCR 先用预训练

优先训练车牌检测模型，因为车牌检测是后续 OCR 的前提。

可以先做：

```text
YOLO plate detector
输入：车辆 crop 或整图
输出：plate box
```

OCR 暂时使用预训练模型，不急着自己训练。

如果检测稳定，但 OCR 在黄牌货车上错误很多，再考虑微调 OCR。

## 6. 黄牌货车的特殊性

本项目识别对象主要是大货车，因此车牌大多是黄色牌照。

这有利也有弊。

有利点：

```text
颜色先验明显，黄牌区域更容易定位
车牌格式相对固定，可以做规则后处理
识别目标更聚焦，不需要覆盖所有小汽车蓝牌场景
```

困难点：

```text
夜间黄牌容易过曝
高速路摄像头下车牌经常模糊
重型货车车头角度变化大
车牌可能被灯光、泥污、栏杆或反光遮挡
```

因此最终系统应该有“可识别”和“不可识别”的分流，而不是强制每张图都输出车牌号。

## 7. 可评价指标

车牌模块可以分开评估。

### 7.1 车牌检测指标

```text
Precision
Recall
mAP50
mAP50-95
```

重点看：

```text
车牌小目标是否漏检
夜间和过曝场景是否漏检
车辆框内检测是否优于整图检测
```

### 7.2 OCR 识别指标

```text
整牌准确率
字符准确率
可读车牌识别率
低置信度拒识准确性
```

对本项目来说，拒识能力很重要：

```text
看不清的车牌不要乱输出
能看清的车牌尽量准确识别
```

## 8. 推荐推进路线

当前最现实的路线是：

```text
第一步：保留车辆检测模型作为前置模块
第二步：收集或标注少量真实黄牌货车车牌框
第三步：训练单类别 YOLO 车牌检测模型
第四步：接入预训练车牌 OCR
第五步：在真实高速货车图片上统计成功、失败和拒识案例
```

如果短期内没有车牌标注，则先做：

```text
车辆检测模块 + 车牌识别预训练 baseline + 失败案例分析
```

这也可以写进项目，因为它体现了：

```text
模块化系统设计
真实场景困难分析
小目标检测问题
低质量图像下的拒识机制
工程可落地的推理流程
```

## 9. 当前阶段结论

车牌模块不是无法做，但不能只依赖现有 chexing 数据集直接训练。

如果没有车牌框和车牌文本标注，最合理的做法是：

```text
先用公开数据和预训练模型建立 baseline
再用少量真实高速黄牌货车图做人工标注和领域适配
最后明确不可识别场景，加入低置信度拒识机制
```

项目表达上，重点不是声称所有车牌都能识别，而是说明：

```text
系统能够在车辆检测后定位车牌
对清晰黄牌进行识别
对过曝、模糊、过小车牌进行低置信度拒识
```

这才符合真实高速路图像的难度。
