# 超载风险预警模块宏观方案

## 1. 模块定位

超载判断不能简单理解为“看一张图片就判断是否超载”。

真实交通执法或工业系统中，超载通常需要结合：

```text
车辆类型
车辆轴数
车辆核定载质量
实际称重数据
车牌与车辆档案
货物状态
历史通行或称重记录
```

单张高速路监控图像只能提供一部分视觉线索，无法可靠得到真实重量。因此本项目中更合理的定位是：

```text
超载风险预警
```

也就是输出：

```text
低风险
中风险
高风险
无法判断
```

而不是直接输出一个绝对结论：

```text
超载 / 不超载
```

这样更符合真实场景，也更容易在科研训练项目中讲清楚技术边界。

## 2. 系统整体流程

最终系统可以设计为：

```text
输入高速路货车图片
  -> 车辆检测与车型分类
  -> 车牌检测与识别
  -> 车辆属性提取
  -> 外部信息关联
  -> 超载风险评分
  -> 输出结构化结果
```

结构化输出示例：

```json
{
  "vehicle_type": "Zhatu",
  "vehicle_conf": 0.91,
  "plate_text": "黄A0A330",
  "plate_conf": 0.86,
  "visual_features": {
    "cargo_visible": true,
    "cargo_height_risk": "medium",
    "rear_suspension_risk": "unknown",
    "axle_count": 6
  },
  "external_features": {
    "rated_load_available": false,
    "historical_weight_available": false
  },
  "overload_risk": "medium",
  "risk_conf": 0.64,
  "reason": "车型为重型货车，货厢载货明显，但缺少称重和车辆核载信息"
}
```

## 3. 可以获取哪些因素

超载风险可以分成三类信息来源。

### 3.1 图像内可获取的信息

这些信息可以从当前图片或模型结果中得到。

```text
车辆类别
车辆位置
车牌位置
车牌颜色
车牌号码
车辆朝向
车厢类型
货物是否可见
货物是否超过车厢边缘
货物堆高程度
是否有明显篷布覆盖
轮轴数量
车辆姿态是否异常
车轮或悬挂是否明显下沉
```

这些信息中，目前最可靠的是：

```text
车辆类别
车辆框
车牌框
部分清晰图像中的车牌号
货厢是否有明显货物
```

较难稳定获取的是：

```text
准确轴数
悬挂下沉程度
真实货物重量
真实车辆总质量
```

### 3.2 车牌关联的信息

如果车牌识别成功，可以用车牌作为索引，关联外部数据。

理论上可以关联：

```text
车辆登记类型
核定载质量
核定总质量
轴数
车辆所有人或企业
历史称重记录
历史超限记录
ETC 或收费站通行记录
动态称重系统记录
```

在真实系统中，这些数据通常来自交通管理、治超站、收费站或企业数据库。

在本科研训练项目中，如果没有真实外部数据库，可以先设计一个模拟表：

```csv
plate,vehicle_type,rated_load_ton,axle_count,history_overload_count
黄A0A330,Zhatu,31,6,2
黄B12345,Sanhuo,18,4,0
```

这样可以证明系统具备“车牌关联车辆档案”的能力。

### 3.3 外部称重或传感器信息

如果未来有称重数据，超载判断会更可靠。

可用外部数据包括：

```text
地磅称重
动态称重 WIM
收费站称重
道路传感器
人工执法记录
企业装载记录
```

这类数据可以直接给出实际重量，是判断超载的关键。

如果没有这类数据，视觉模块只能做风险预警，不能做严格执法级判断。

## 4. 如何和车辆识别结果关联

车辆识别模块已经能输出五类货车：

```text
Kong
Sanhuo
Jizhuangxiang
Caoguan
Zhatu
```

这些类别可以作为超载风险的先验信息。

例如：

```text
Kong：空车或低载风险更高概率
Sanhuo：货物形态多样，需要看车厢载货状态
Jizhuangxiang：货物在箱体内，单图难判断重量
Caoguan：需要关注罐体类型，但视觉难估计真实载重
Zhatu：渣土/散装货物可见时，超载风险更容易从外观判断
```

因此可以设计一个规则融合模块：

```text
车型类别
  + 货物可见性
  + 车厢满载程度
  + 车牌是否识别成功
  + 是否有关联档案
  + 是否有历史超载记录
  -> 超载风险等级
```

这比单纯从图片直接判断“超载/不超载”更可信。

## 5. 推荐的技术路线

### 5.1 第一阶段：规则型风险评分

先不训练超载模型，做一个可解释的规则评分模块。

示例：

```text
基础分 = 0

如果车型是 Zhatu 或 Sanhuo：+2
如果检测到货物明显高于车厢：+3
如果车厢接近满载：+2
如果车牌识别失败：+1
如果历史超载次数 > 0：+2
如果存在实际称重且超过核载：直接高风险
```

输出：

```text
0-2：低风险
3-5：中风险
6 以上：高风险
缺少关键视觉信息：无法判断
```

优点：

```text
实现快
可解释
适合作为科研训练系统原型
可以和车辆识别、车牌识别模块串起来
```

缺点：

```text
不是严格意义上的超载识别模型
依赖规则设计
不能替代真实称重
```

### 5.2 第二阶段：弱监督风险分类

如果 overload 数据集本身有超载/未超载标签，可以训练一个图像分类模型：

```text
输入：车辆 crop
输出：低风险 / 中风险 / 高风险 或 超载 / 未超载
```

但需要注意：

```text
标签是否真实来自称重
图片是否能反映载重
类别是否均衡
是否存在拍摄角度和场景偏差
```

如果标签只是人工从外观判断，就应该称为“视觉风险分类”，不要称为真实超载识别。

### 5.3 第三阶段：多源融合模型

如果未来有车牌档案、历史称重、车辆核载数据，可以做多源融合。

输入特征包括：

```text
视觉特征
车型类别
车牌识别结果
车辆核载信息
历史称重信息
历史违规信息
```

模型可以是：

```text
规则系统
逻辑回归
随机森林
XGBoost
轻量 MLP
```

对于科研训练项目来说，规则系统或树模型已经足够解释清楚。

## 6. 当前没有外部数据时能做什么

如果没有车牌档案、称重记录和真实超载标签，仍然可以做一个有深度的模块：

```text
1. 使用车辆检测模型得到车辆类别和车辆框
2. 设计车牌识别接口，能识别则作为关联字段，不能识别则返回 low confidence
3. 基于车型和视觉线索做超载风险评分
4. 明确输出风险等级和不可判断原因
5. 在文档中说明视觉判断的边界
```

这样系统不是假装解决真实称重问题，而是做：

```text
面向高速货车图像的超载风险辅助预警
```

这个定位更合理，也更适合写进简历。

## 7. 简历中可以强调的深度

项目可以这样表达技术深度：

```text
构建了车辆检测、车牌识别和超载风险预警的多模块推理流程
针对高速货车图像中车牌小、反光强、遮挡多的问题，设计低置信度拒识机制
将车型识别结果作为超载风险评估的先验信息
设计车牌到车辆档案/历史称重数据的关联接口
将超载判断从单图像二分类转化为多源信息融合的风险评分问题
```

不要写：

```text
实现了精准超载检测
仅凭图像准确判断车辆是否超载
```

更稳妥的写法是：

```text
实现面向高速货车场景的车辆检测与车型分类，并设计车牌识别和超载风险辅助预警模块，支持多源信息融合和低置信度拒识。
```

## 8. 当前推荐推进顺序

当前最现实的推进顺序是：

```text
1. 车辆检测与车型分类模块收尾
2. 使用已有车牌检测权重做车牌检测 baseline
3. OCR 暂时接预训练模型或先保留接口
4. 设计超载风险评分规则和输出格式
5. 用几张真实样例跑通完整后端流程
6. 后续有时间再补车牌标注和 OCR 微调
```

这样可以最大化项目完成度，同时避免陷入大规模标注和不可验证的超载判断。
