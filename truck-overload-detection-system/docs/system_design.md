# 高速道路重载货车超载嫌疑智能识别系统设计说明

## 一、系统背景

高速道路货运车辆通行量大，部分重载货车可能存在严重超载风险。传统执法依赖人工巡查、经验判断和后续称重复核，存在发现不及时、筛查效率低等问题。

本系统面向交通管理部门、高速交警和道路执法场景，通过导入高速道路监控图片或现场取证图片，对图片中的车辆进行目标检测，输出车辆类别、置信度和检测框，为后续超载嫌疑识别、车牌识别、现场拦截和复核称重提供软件基础。

## 二、总体目标

系统目标是构建一个可运行、可扩展、便于接入视觉模型的软件平台。当前软件著作权原型阶段重点完成以下能力：

1. 支持车辆图片上传和预览；
2. 后端保存上传图片，形成可追溯记录；
3. 调用当前已训练的 YOLO 目标检测模型；
4. 前端清晰展示检测结果、标注图和模块接入状态；
5. 为超载嫌疑识别和车牌识别预留真实模型接入接口，便于后续升级。

## 三、系统架构

系统采用 Vue + FastAPI 的前后端分离思想，并支持一体化部署。开发阶段 Vue 由 Vite 单独启动，接口请求代理到 FastAPI；正式打包后，FastAPI 可以直接托管 Vue 构建产物，使软件界面、接口服务和模型调用模块保存在同一个软著项目中。

```text
用户浏览器
    ↓ 上传图片
Vue 前端页面
    ↓ HTTP 请求
FastAPI 后端服务
    ↓ 保存图片
uploads 图片归档目录
    ↓ 调用业务模块
目标检测服务 / 车型派生服务 / 超载预留服务 / 车牌预留服务
    ↓ 汇总结果
JSON 返回前端展示
```

## 四、模块划分

### 1. Vue 前端展示模块

文件位置：

```text
frontend/index.html
frontend/src/App.vue
frontend/src/main.js
frontend/src/api/vehicleAnalysis.js
frontend/src/components/ResultCard.vue
frontend/src/styles.css
```

主要职责：

- 提供图片选择和拖拽上传入口；
- 显示图片预览；
- 通过 API 模块调用后端识别接口；
- 展示目标检测结果、车辆类别、置信度和坐标框；
- 展示超载和车牌模块的待接入状态；
- 对需要人工复核的结果进行醒目标记。

采用 Vue 后，后续可继续扩展登录页、历史记录页、模型配置页、批量识别页、执法处置记录页等模块。

### 2. 后端 API 模块

文件位置：

```text
backend/main.py
```

主要职责：

- 提供首页访问；
- 提供健康检查接口；
- 提供 `/api/analyze` 图片分析接口；
- 校验上传文件类型；
- 保存上传图片；
- 调用目标检测服务并汇总各模块状态。

### 3. 目标检测模块

文件位置：

```text
backend/services/vehicle_detector.py
```

当前已接入用户训练得到的 YOLO 模型权重：

```text
/home/kang/research/src/models/runs/detect/BIT_Vehicle_Model/weights/best.pt
```

该模块主要输出：

- 检测目标列表；
- 类别英文名和中文名；
- 检测置信度；
- 检测框坐标；
- 带框标注图路径。

数据集类别来自 `data/yolo_dataset/bit_vehicle.yaml`，包括 Bus、Microbus、Minivan、Sedan、SUV、Truck。

目标检测模型不是只能识别训练集原图，但会受到训练数据分布影响。若输入图片与训练集在视角、距离、清晰度、遮挡、车型外观或道路环境上差异较大，模型置信度可能下降，甚至低于检测阈值而不显示结果。系统前端提供置信度阈值调节，用于在“提高召回率”和“减少误检”之间做取舍。

### 4. 超载嫌疑识别模块

文件位置：

```text
backend/services/overload_detector.py
```

当前阶段尚未接入超载嫌疑判定模型，因此该模块只返回 `pending` 状态和人工复核提示，不输出虚假的超载结论。后续可接入图像分类模型、目标检测模型或结合车辆姿态和载荷状态的综合判断模型。

建议输出：

- 是否疑似严重超载；
- 超载嫌疑置信度；
- 风险等级；
- 判断依据。

### 5. 车辆类型识别模块

文件位置：

```text
backend/services/vehicle_type_classifier.py
```

当前车型结果从目标检测模型的类别输出中派生，例如 Truck 显示为“货车”。它不是独立的重型/中型/罐式/厢式细分类模型。后续如果训练了细粒度车型分类模型，可在该模块中替换为独立推理。

建议输出：

- 车辆类型；
- 车型识别置信度。

### 6. 车牌识别模块

文件位置：

```text
backend/services/plate_recognizer.py
```

当前阶段尚未接入车牌识别模型，因此该模块只返回 `pending` 状态。真实系统中通常包括两个步骤：

```text
车牌区域检测 -> 车牌字符识别
```

可选方案包括 OpenCV 定位、PaddleOCR、EasyOCR 或自训练车牌识别模型。

### 7. 模型加载模块

文件位置：

```text
backend/models/model_loader.py
```

该模块统一管理模型配置和加载逻辑，使业务服务不直接绑定具体模型框架。后续可以根据 `backend` 类型加载 PyTorch、ONNX Runtime 或 OpenCV 模型。

## 五、模型接入方式

当前已接入的目标检测模块核心流程如下：

```python
def predict(self, image_path: Path, annotated_dir: Path) -> dict:
    model = YOLO(str(self.weights_path))
    results = model.predict(source=str(image_path), conf=0.5)
    return {
        "module_status": "connected",
        "detections": detections,
        "annotated_image_url": annotated_url,
    }
```

后续接入真实模型时，只需要替换 `predict` 内部逻辑：

```text
读取图片 -> 图像预处理 -> 模型推理 -> 后处理 -> 返回统一字段
```

只要保持 `detection`、`vehicle_type`、`overload`、`plate` 四类模块化返回结构不变，前端页面和后端 API 汇总逻辑就不需要大改。

## 六、业务流程

1. 执法人员打开系统页面；
2. 上传高速道路监控图片；
3. 系统保存图片到后端目录；
4. 系统调用目标检测模型；
5. 前端展示检测框、车辆类别、置信度和模块状态；
6. 对超载和车牌识别显示待接入状态；
7. 交警或开发人员根据检测结果进行人工复核或后续模型接入。

## 八、软著材料表述建议

本系统以高速道路车辆图像为输入，面向交通执法辅助场景，当前完成车辆目标检测、检测结果可视化和模块化接口管理。系统采用 Vue + FastAPI 架构，将前端交互、后端接口、图片归档、YOLO 模型调用和后续模型预留接口进行分层封装，提高了软件可维护性和模型扩展能力。当前原型已实现目标检测输入输出流程，后续可接入超载嫌疑判定模型和车牌识别模型进一步完善实际应用效果。
